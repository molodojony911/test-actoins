# Правила деплоя

Деплой выполняется автоматически через GitHub Actions (файл `.github/workflows/build-deploy.yml`): образ собирается, публикуется в **GitHub Container Registry (GHCR)**, затем по **SSH** обновляется контейнер на сервере.

## Когда запускается

1. **Push в ветку `main`** — полный цикл: сборка → push в GHCR → деплой.
2. **Вручную** — в GitHub: **Actions** → workflow **Build, push to GHCR, deploy** → **Run workflow** (`workflow_dispatch`).

Деплой **не** выполняется для других веток и для pull request (если не добавить отдельные триггеры).

## Требования к серверу

- Установлен **Docker**, пользователь из `SSH_USER` может выполнять `docker` (например, в группе `docker` или через rootless — как настроено у вас).
- Доступ по **SSH** с ключом: публичная часть ключа добавлена в `~/.ssh/authorized_keys` пользователя на сервере.
- Порт **SSH** по умолчанию **22**. Для другого порта в workflow в шаг `appleboy/ssh-action` нужно добавить параметр `port: ${{ secrets.SSH_PORT }}` и завести секрет `SSH_PORT`.
- На сервере должен быть свободен порт приложения (см. ниже `APP_PORT`, по умолчанию **8000**).

## Секреты и переменные репозитория

Настройка: **Settings** → **Secrets and variables** → **Actions**.

**Важно:** workflow должен быть **закоммичен и запушен** в ту ветку, откуда он запускается (у вас — `main`). Если в логе у шага `Run remote deploy` в блоке `with:` указано **`key:`**, а не **`key_path:`**, на GitHub всё ещё **старая** версия файла — подтяните актуальный `.github/workflows/build-deploy.yml`.

| Имя | Тип | Обязательно | Назначение |
|-----|-----|-------------|------------|
| `SSH_HOST` | Secret **или** Variable | да | Хост сервера (IP или домен). В workflow: сначала Variable `SSH_HOST`, иначе Secret. |
| `SSH_USER` | **Variable (рекомендуется)** или Secret | да | Пользователь SSH (`ubuntu`, `root`, …). Имя пользователя не секрет: задайте **Repository variable** `SSH_USER` — так проще отладить «username is empty». Если используете только Secret, имя должно быть в точности **`SSH_USER`**. |
| `SSH_PRIVATE_KEY` | Secret | да | Приватный ключ целиком (PEM, строки `BEGIN` / `END`). |
| `GHCR_USERNAME` | Secret | да | Логин для `docker login ghcr.io` на сервере (обычно GitHub username владельца PAT). |
| `GHCR_PULL_TOKEN` | Secret | да | Personal Access Token с правом **`read:packages`**. Для приватного пакета в GHCR может понадобиться расширенный доступ к репозиторию пакета. |
| `SSH_PORT` | Secret | нет | Только если SSH не на порту 22 и вы добавили `port` в workflow. |

Секреты не коммитить в репозиторий; хранить только в GitHub Secrets.

## Образ в GHCR

- Имя образа: `ghcr.io/<владелец>/<репозиторий>` в **нижнем регистре** (как у GitHub).
- Теги: **`latest`** и **SHA коммита** (для отката можно на сервере вручную указать образ с тегом по SHA).
- Первый push: убедитесь, что пакет в GHCR **доступен** пользователю PAT (публичный пакет или выдан доступ **read** на пакет для пользователя/организации).

## Имя контейнера и порт

Задаются в workflow в блоке **`env`**:

- `DEPLOY_CONTAINER_NAME` — имя контейнера на сервере (по умолчанию `test-gitact`).
- `APP_PORT` — порт хоста и контейнера (по умолчанию `8000`, согласован с `Dockerfile` / `EXPOSE`).

При смене порта или имени контейнера править **только** этот блок в `.github/workflows/build-deploy.yml`, затем закоммитить в `main`.

## Порядок операций на сервере (деплой)

1. `docker login ghcr.io` с использованием `GHCR_USERNAME` и `GHCR_PULL_TOKEN`.
2. `docker pull` образа с тегом **`latest`**.
3. Остановка и удаление контейнера с именем `DEPLOY_CONTAINER_NAME` (если существует).
4. Запуск нового контейнера: detached, `--restart unless-stopped`, проброс `-p APP_PORT:APP_PORT`.

## Правила для команды

1. **Продакшен-деплой** считается выполненным после успешного прохождения workflow на `main` (или после ручного запуска с нужной веткой — сейчас триггер только `main`).
2. Не отключать и не ослаблять проверки без согласования: падающая сборка не должна маскироваться «ручным» деплоем старого образа без понимания причины.
3. Менять секреты и ключи SSH только через **Settings** репозитория; после смены ключа обновить `authorized_keys` на сервере.
4. PAT для `GHCR_PULL_TOKEN` хранить с минимальными правами (`read:packages`); при компрометации — отозвать токен и создать новый, обновить секрет.

## Устранение неполадок SSH в Actions

### `ssh.ParsePrivateKey: ssh: no key found` / `no supported methods remain`

Это почти всегда значит, что **приватный ключ не распознан** (не тот формат, обрезан, пустой секрет) или **не подставился целиком**.

1. В секрет `SSH_PRIVATE_KEY` нужно вставить **именно приватный** ключ (строки вида `-----BEGIN OPENSSH PRIVATE KEY-----` … `-----END …-----` или `BEGIN RSA PRIVATE KEY`), а не публичный `.pub`.
2. Копируйте ключ **целиком**, с первой и последней строкой, без лишних пробелов в начале строки `BEGIN` и без кавычек вокруг всего ключа в UI GitHub.
3. Если ключ **с паролем (passphrase)**, в шаг `appleboy/ssh-action` нужно добавить параметр `passphrase: ${{ secrets.SSH_KEY_PASSPHRASE }}` и завести секрет с паролем; либо сгенерировать отдельный ключ **без** passphrase только для деплоя.
4. В workflow ключ записывается в файл через переменную окружения (шаг **Write SSH private key**), чтобы многострочный PEM не портился при передаче в action.

После смены ключа не забудьте положить **соответствующую публичную** строку в `~/.ssh/authorized_keys` на сервере для `SSH_USER`.

## Локальная проверка образа

Сборка и запуск без GitHub:

```bash
docker build -t test-gitact-local .
docker run --rm -p 8000:8000 test-gitact-local
```

Проверка API: `http://127.0.0.1:8000/docs`.
